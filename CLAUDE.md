# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single-binary Go WAF + reverse proxy: embedded Coraza v3 (OWASP CRS) for request inspection, Echo for HTTP/routing, SQLite (modernc, pure Go, no CGO) for all storage, and an HTMX + Tailwind admin dashboard served from the same binary.

```
[Client] → [Coraza WAF + Proxy (Echo)] → [Backend App(s)]
                     ↕
               [SQLite DB]
                     ↕
              [HTMX/Tailwind UI]
```

## Commands

Requires **Go 1.25+**.

```bash
make build       # go generate (minifies JS) + go build -> ./coraza-waf-mod
make generate    # regenerate minified JS only (go generate ./...), no build
make run         # build + run
make test        # go test ./...
make lint        # golangci-lint run ./... (config: .golangci.yml, pinned version in LINT_VERSION)
make hooks       # git config core.hooksPath .githooks, enabling the pre-commit lint hook — a build/test prerequisite, so this runs automatically; rarely needs calling by name
make clean       # remove binary, minified JS, dist/
make dist        # cross-compile linux/amd64 + linux/arm64 + windows/amd64 release binaries (CGO_ENABLED=0)
make checksums   # sha256sum dist/* -> dist/checksums.txt
make tag VERSION=v1.0.0  # annotated git tag + push to origin, triggers the GitHub Actions release workflow
```

Run a single test: `go test ./internal/proxy/ -run TestName -v` (substitute the package). Existing tests live in `internal/proxy/`, `internal/security/ratelimit/`, `internal/security/ja3/`, `internal/security/ja4/`, `internal/security/geo/`, `internal/security/asn/`, `internal/storage/`, `internal/notify/mailer/`, `internal/notify/accesslog/`, `internal/security/autoban/`, `internal/security/threatscore/`, `internal/security/adaptive/`, `internal/security/challenge/`, `internal/security/totp/`, `internal/services/`, `internal/security/waf/`, and `internal/ui/`. Run `gofmt` and `make lint` on changed Go files; the existing tests are table-driven (e.g. `TestRegistryMatchPrefixPriority`), so add cases next to the package being changed in that style.

**Linting** (`.golangci.yml`, golangci-lint v2 config format): `linters.default: standard` (errcheck, govet, ineffassign, staticcheck, unused) plus the `common-false-positives`/`std-error-handling` exclusion presets, which cover the usual `defer f.Close()`-style false positives. What those presets don't catch — e.g. `(*flag.FlagSet).Parse` on a set built with `flag.ExitOnError` (a checked error there is genuinely unreachable code, since a parse failure calls `os.Exit` before `Parse` returns) — is suppressed inline with a same-line `//nolint` and a short reason, matching the handful of pre-existing `//nolint` comments in `internal/storage/db.go` (idempotent `ALTER TABLE` migrations, an intentionally-ignored stats `Scan`). `.github/workflows/ci.yml`'s `lint` job runs `golangci/golangci-lint-action` pinned to the same version as the Makefile's `LINT_VERSION`, and `release` now `needs: [lint, test]` — a lint failure blocks a tag release exactly like a test failure does. `.githooks/pre-commit` runs the identical `golangci-lint run ./...` locally before every commit once something has pointed git at it (`git config core.hooksPath .githooks` — plain `.git/hooks/` isn't version-controlled, so this is the only way to ship a hook to every clone; there's no npm-postinstall equivalent here by design, so `hooks` is instead a prerequisite of the `build`/`test` Makefile targets — idempotent and silent once already set — so it gets enabled the first time anyone runs either, not just via an easily-forgotten standalone `make hooks`); `git commit --no-verify` skips it for one commit.

**One-shot subcommands** (exit after completing):

```bash
coraza-waf-mod prune [--db path] [--retention days] [--vacuum]
  # Delete request log rows older than --retention days (default 30) + expired admin sessions; see main.go:runPruneOnly
  # --vacuum then rebuilds the DB file (VACUUM + WAL truncate) — DELETE alone never shrinks a SQLite file

coraza-waf-mod setup --db waf.db --admin-email e@x.com [--domain d] [--acme-email e]
  # Seed admin credentials and optional ACME config into the DB (reads password from stdin).
  # Idempotent for credentials — safe to re-run on upgrade without overwriting a changed password.

coraza-waf-mod gencert --cert cert.pem --key key.pem --hosts ip1,hostname [--days 3650]
  # Generate a self-signed ECDSA P-256 cert with the given SANs (hostnames or IPs).

coraza-waf-mod --version
  # Print the version baked in via -ldflags "-X main.version=..." (make sets it from git describe; bare go build gives "dev").
```

**Never run `go build` directly after editing JS** — `go:generate` (which runs the JS minifier in `tools/minify`) only fires via `make build`/`make dist`, not bare `go build`. If you must, run `go generate ./...` first.

Do not start the server (`go run .`, `make run`, etc.) without the user explicitly asking each time — verify changes with throwaway `go test` files instead, and delete them once they pass.

## Architecture

**Request flow** (`internal/proxy/handler.go`, single `Handle` method, in order): IP blocklist check → geo blocklist check → bot challenge gate → global rate limit → per-service rate limit → Coraza WAF inspection → reverse-proxy to the matched backend. IP and geo blocklist are checked _before_ the challenge gate — an explicit admin/autoban block (or a blocked country) has nothing left to prove by solving a JS challenge first; when the challenge gate ran first, a banned IP with no (or an expired/cleared) bypass cookie just kept getting redirected to `/_cz/challenge` instead of the expected 403, logged as repeated unsolved `bot_challenge` attempts rather than `ip_blocked` — which reads to an admin as "the ban isn't taking effect" even though the `ip_rules` entry is correct. The WAF check still runs _after_ the challenge gate, deliberately: see `autoban`'s own comment — a brand-new attacker with no block rule yet needs to accumulate unsolved-challenge points before autoban ever creates one, and that bootstrapping only works if the challenge gate can intercept it before a WAF verdict exists. Every stage logs to SQLite via a fire-and-forget queue, never blocking the hot path. Every response (blocked or proxied) gets its `Server` header forced to `Coraza WAF Mod`, overwriting whatever the backend sent — set independently in both `internal/proxy/handler.go` and `internal/services/registry.go`'s `ModifyResponse` since proxied responses pass through the latter.

Request logs are not kept forever, but pruning is **not** automatic inside the running server — `coraza-waf-mod prune [--db path] [--retention days]` (`runPruneOnly` in `main.go`) is a separate one-shot CLI mode that opens the DB, deletes rows older than `--retention` days (default 30; `<= 0` disables pruning), and exits without starting the WAF/proxy/admin UI. It's meant to be invoked by an external scheduler — cron, or the systemd timer in `deploy/coraza-waf-mod-prune.{service,timer}` — so a multi-second delete never shares the live process's DB connection pool with request traffic. `storage.DB.PruneOldRequests` itself deletes in batches of `pruneBatchSize` (2000) rows with a short pause between batches rather than one giant `DELETE`, since SQLite holds its single process-wide write lock for the full duration of an unbatched delete. Deleting never shrinks the file, though — SQLite only free-lists the pages for reuse — so `prune --vacuum` (which `install.sh`'s 15-day prune timer passes) additionally runs `storage.DB.Vacuum` afterwards: a full `VACUUM` rebuild followed by `PRAGMA wal_checkpoint(TRUNCATE)`, without which the reclaimed space just migrates into a ballooned `-wal` sidecar until the last connection closes. The vacuum deliberately lives only in this one-shot mode, never the running server: `VACUUM` is one long write transaction, and run from a separate process the live server's writers merely wait out their `busy_timeout` instead of the rebuild monopolizing the serving pool.

**Services are DB-backed, not config-driven.** `storage.MigrateConfigApps` exists to seed the `services` table from a legacy `apps:` list on first-ever startup, but `main.go` now calls it with `nil` (`db.MigrateConfigApps(nil)`) — there is no YAML file read anywhere in the running server, so in practice this is always a no-op today; `/admin/services` in the UI is the sole source of truth for routing. `services.Registry` (`internal/services/registry.go`) holds the current service list plus one pre-built `httputil.ReverseProxy` per service behind a `sync.RWMutex`, and is rebuilt wholesale via `Reload(db)` whenever a service is added/edited/removed — no restart needed. All proxies share one `backendTransport` with explicit dial (5s) and response-header (10s) timeouts, since Go's default transport has no upper bound on waiting for a backend and will otherwise stall browser connection slots. `services.Validate` (called before a new service is saved and before `Probe` ever contacts it) rejects cloud instance-metadata endpoints (`169.254.169.254`, AWS IMDSv2's IPv6 `fd00:ec2::254`, `metadata.google.internal`) by literal host and by resolved IP (CodeQL `go/request-forgery`) — deliberately narrow, since arbitrary internal/private backends are this field's expected, legitimate use and are not blocked wholesale.

**Routing precedence** in `Registry.Match(host, path)`: all Prefix-matched services are checked first (longest prefix wins), then Host matches — mirroring nginx's `location` blocks beating `server_name` defaults. For prefix-matched services, `internal/proxy/handler.go`'s `Handle()` strips the matched prefix from the request path before proxying (restored before logging, so the admin UI shows the real client path) — like `proxy_pass http://backend/` in nginx. Host-matched services are passed through untouched (virtual hosting).

**Varnish cache is a loopback sandwich, and its VCL is static.** The flow for a service with `cache_enabled` set (the "Cache" toggle on `/admin/services`) is client → WAF (full pipeline) → varnishd (`127.0.0.1:6081`) → WAF cache-return listener (`127.0.0.1:6082`, `startCacheReturn` in `main.go`, always running) → real backend. `Registry.Reload` re-targets the service's outer `ReverseProxy` at Varnish and also builds a straight-to-backend twin in `Registry.direct` (target deliberately stripped to scheme+host — the outer director already joined any backend base path, joining twice would double it); `Registry.CacheReturnHandler` routes miss fetches to that twin by the `X-Cache-Service` header the outer Director sets. Because backend selection happens back in the WAF, `deploy/varnish/default.vcl` has exactly one permanent backend (`127.0.0.1:6082`) and **never changes when services do** — no VCL edits, no `systemctl reload`, everything is UI-driven; `install.sh` installs varnish (best-effort) and writes the VCL + a loopback-only systemd override. Config is `storage.VarnishConfig` (meta keys `varnish_enabled`/`varnish_addr`/`varnish_return_addr`; the return address is not exposed in the UI and must match the VCL). Cache-poisoning defenses: the outer Director scrubs `spoofableHostHeaders` (`X-Forwarded-Host`, `X-Original-URL`, etc.) and overwrites `X-Cache-Service`; `SaveVarnishConfig` rejects non-loopback Varnish addresses; `startCacheReturn` refuses non-loopback binds; the return handler 403s non-loopback peers; the VCL partitions the hash by `X-Cache-Service` and 403s non-local clients. Global switch and per-service flags apply via `Registry.Reload` with no restart — with the switch off, cache-flagged services proxy direct to their backends. Session-aware caching (`CacheBySession`/`SessionCookieName`, off by default) is an opt-in per-service toggle that partitions the cache by the _hash_ of a named session cookie's value — sent as `X-Cache-Session` — instead of Varnish's default behavior of refusing to cache any cookie-bearing request; the cookie value is hashed (never forwarded raw) to keep the header short and avoid putting a live session token into Varnish's own logs. Per-service TTL floor/ceiling and grace/keep are admin-configurable ("Cache tuning" on `/admin/services`, `storage.Service.CacheTTLFloor/CacheTTLCeiling/CacheGrace/CacheKeep`, `db.SetServiceCacheTuning`) — the Director sends them as `X-Cache-TTL-Floor`/`-Ceiling`/`-Grace`/`-Keep` headers (plain seconds), and the VCL parses them with `std.duration()` in `vcl_backend_response`, falling back to its own defaults (1h floor, no ceiling, 30s grace/keep) when a header is absent; the VCL text still never changes per service, only the headers sent to it do. `services.Purge(vcfg, serviceName)` invalidates a service's cached objects by sending a `PURGE` request straight to Varnish's client port over loopback (same trust model as the cache-return listener) — the VCL `ban()`s every object tagged `X-Cache-Service == serviceName` (tagged in `vcl_backend_response`, stripped from client responses in `vcl_deliver`); `Purge` validates the service name against a conservative charset (`safeServiceNameForBan`) first since Varnish's ban grammar has no escaping mechanism for the compared value. Wired to the "Purge" button on `/admin/services` via `POST /admin/services/cache-purge/:id`.

**Health is passive, not polled.** There is no background health-check loop; `Registry.markHealth` is driven entirely by real proxied traffic outcomes via each `ReverseProxy`'s `ErrorHandler` (failure → unhealthy) and `ModifyResponse` (response received → healthy) hooks. The one exception is `services.Probe()`, a single one-shot reachability GET run only when a service is first added through the UI, to reject obviously-dead backends before they're saved.

**TLS**: `--listen-tls` activates HTTPS; ACME email, per-service certs, and cert mode are all stored in the DB. `startTLS` in `main.go` builds an `autocert.Manager` if an email is stored, then starts HTTP (ACME challenge handler + redirect) and HTTPS. Each service can have its own TLS via the admin UI: an uploaded custom cert, or per-service auto-issue. `Registry.GetCertificateFunc` resolves certs by SNI: per-service custom cert → per-service/legacy autocert (if host is in `Registry.HostPolicy()`) → global fallback. Uploaded private keys are stored on disk under `certs/services/<name>/` (mode 0600), not in `waf.db`.

**WAF engine** (`internal/security/waf/engine.go`): directive order matters and has bitten this project before — `@coraza.conf-recommended` sets `SecRuleEngine DetectionOnly` as a safe default, so any `SecRuleEngine On` override must be placed _after_ the CRS includes in the directives string, or the WAF silently scores every rule but blocks nothing.

**SQLite concurrency** (`internal/storage/db.go`): PRAGMAs (`busy_timeout`, `journal_mode=WAL`, `synchronous=NORMAL`) are passed via DSN query params (`_pragma=...`), not a post-open `Exec` — with `modernc.org/sqlite`, only DSN params apply to every pooled connection; a bare `Exec` only configures whichever single connection happens to run it. `SetMaxOpenConns(8)` is safe because WAL lets readers proceed concurrently with the one writer SQLite always serializes. Request logging never runs synchronously on the request path: `DB.QueueRequest` pushes onto a buffered channel (`logQueue`, cap 10000) drained by one dedicated `runLogWorker` goroutine; `DB.Close()` drains the queue before closing the connection.

**Admin UI** (`internal/ui/handlers.go`, `internal/ui/templates/`): one Echo group mounted at `cfg.Admin.Path` (default `/admin`) behind **session-cookie auth** (bcrypt password check, 24h cookie with `Secure` set on HTTPS, sessions stored in the `sessions` SQLite table — not HTTP Basic Auth). Login is optionally two-step: with TOTP 2FA enabled (the "Two-Factor Authentication" card on Settings), a correct password parks the login behind a short-lived `cz_2fa` cookie (`internal/ui/twofactor.go`, in-memory pending store, 5-min TTL) and asks for a 6-digit code — an RFC 6238 authenticator code (`internal/security/totp`, pure Go, verified locally), a one-time backup code (SHA-256 hashes in the `admin_totp_backup_codes` meta key, handed out once at enrollment), or an emailed single-use recovery code sent through the existing Cloudflare mailer (`Reporter.SendLoginCode`; offered only when email is configured, 60s resend throttle) — before any session cookie is issued. Accepted TOTP counters persist in `admin_totp_last_counter` so a code can't be replayed within its 30s window; enrollment requires a working code before 2FA is enforced (pending secret in `admin_totp_pending_secret`, promoted by `EnableTOTP`), and disabling requires a current code, so a hijacked session alone can't remove the second factor. Login is brute-force-throttled per client IP (`internal/ui/loginlimit.go`: 5 failures in 15 min → 15-min lockout, HTTP 429), and failed 2FA codes feed the same limiter — deliberately _not_ reset on a correct password while 2FA is pending, or re-posting the (already-known) password would launder code-guessing attempts; the IP is the socket peer unless the connection comes from a `--trusted-proxies` CIDR — never echo's `c.RealIP()`, which trusts `X-Forwarded-For` unconditionally and would let an attacker rotate fake IPs past the lockout. All non-GET admin routes additionally require a **CSRF token** (`internal/ui/csrf.go`): the token is derived statelessly as SHA-256 of the session cookie value and enforced by the `csrfProtect` group middleware. HTMX requests pick it up automatically via `hx-headers` on `<body>` in `base.html` (HTMX inherits the attribute), so new `hx-post`/`hx-delete` forms need nothing; plain non-HTMX `method="POST"` forms must include `<input type="hidden" name="_csrf" value="{{.CSRF}}">`, and raw `fetch` POSTs must send `X-CSRF-Token: document.body.dataset.csrf`. Anything that returns sensitive data (e.g. `/settings/backup`, which streams the whole DB) must be a POST, never a GET — SameSite=Lax still sends cookies on top-level cross-site GET navigation. Pages are server-rendered HTML templates with HTMX partial swaps (`renderPartial`) for live updates — e.g. the Logs page tails new rows over SSE (`/admin/logs/stream`, `/admin/api/notifications/stream`) via `ui.LogBroadcaster`, but only when no filters are active; filtered views page through SQLite directly. IP Rules is paginated the same way (`ipRulesPageSize` in `internal/ui/handlers.go`, `storage.DB.ListIPRulesPaginated`'s COUNT-then-`LIMIT`/`OFFSET` shape mirrors `ListRequestsFiltered`'s) — the table can grow into the thousands via autoban, so `IPRulesPage`/`AddIPRule`/`DeleteIPRule` all share `ipRulesRowsData(page)` rather than ever loading the whole table; `GET /admin/ip-rules/rows` (mirroring the existing `/admin/services/rows` pattern) serves the Prev/Next buttons' `hx-get` without a full page reload, and deleting the last row on a page clamps back to the new last page instead of rendering empty. The "Rules overview" block/allow percentages come from a separate `CountIPRulesByType` query against the whole table, not the current page's rows. One gotcha the pagination data map has to route around: `h.render`'s `data["Page"] = page` line (`internal/ui/handlers.go`) unconditionally sets `"Page"` to the _template name_ string for nav highlighting — a pagination page _number_ under that same key gets silently clobbered, so `ipRulesRowsData` (and the Logs page before it) uses `"CurPage"` instead. Frontend JS source lives in `static/js/src/*.js` and is minified by the pure-Go tool in `tools/minify` (via `go:generate`) into `static/js/dist/*.min.js`, which is `//go:embed`-ed (`assets.go`) and served at `/admin/static/js/` — there is no Node/npm toolchain in this repo. Multi-step wizards with hidden (`display:none`) steps, like the service-add form in `internal/ui/templates/services.html`, use `novalidate` on the `<form>` plus manual JS `.value.trim()` checks instead of HTML5 `required`/`checkValidity()` — a `required` input inside a hidden step still participates in native constraint validation per spec but can't be focused for the validation bubble, so Chrome silently blocks submission with no visible error.

**Styling is Tailwind v4 via the browser CDN build, not a build pipeline.** `internal/ui/templates/base.html` loads `<script src="https://cdn.jsdelivr.net/npm/@tailwindcss/browser@4">` and defines the theme (custom `brand`/`navy`/`canvas`/`surface`/`line` colors, `sans`/`mono` font families) as `--color-*`/`--font-*` tokens in a `@theme` block inside `<style type="text/tailwindcss">` — v4's CSS-first config; there is no `tailwind.config.js` / inline `tailwind.config` object, no PostCSS, no purge step (`login.html`, being standalone, carries its own copy of the same setup). That style block also has an `@layer base` restoring two v3 preflight behaviors the UI was built on: default border color (`--color-line` instead of v4's `currentColor`) and `cursor: pointer` on enabled buttons. Utility names follow the v4 scale — `rounded-sm` (4px badges), `shadow-xs`, `outline-hidden`, `bg-linear-to-*`, `placeholder:text-*` — so never reintroduce the removed v3 spellings (bare `rounded`/`shadow`, `outline-none`, `bg-gradient-to-*`, `placeholder-{color}`); v4-only features like the `starting:` variant (`@starting-style` entrances, used on the dashboard right-panel cards and enforcement meters) are fine and preferred over new JS animation. All templates use `class="..."` utilities, including arbitrary-value syntax (`w-[34px]`, `bg-[#E4F5EC]`) for one-off values outside the theme scale; do not introduce `style="..."` attributes or `<style>` blocks for anything Tailwind can express. JS that toggles visibility/color (`static/js/src/*.js`) does it via `classList`, never `element.style.*` — for show/hide, keep a permanent `flex`/`block` class alongside the toggled `hidden` class (Tailwind doesn't merge a bare `hidden` removal back into a layout display value on its own). Go HTML templates: content outside `{{define}}` blocks is never rendered — modal HTML must be inside the `{{define "content"}}` block, not after its closing `{{end}}`. An admin-UI visual redesign (flat/tight look: smaller radii, visible borders, darker `brand.dark`, tighter spacing; login page explicitly untouched) is planned in `UI_REDESIGN_PLAN.md` at the repo root — read it before restyling anything under `internal/ui/templates/` or `static/js/src/`, and update its status line if you execute or change the plan.

**REST API** (`internal/ui/api.go`): `{admin.path}/api/v1/*` is a separate Echo group from the session-cookie admin group — sibling, not nested — authenticated via `Authorization: Bearer <key>` instead of the session cookie, and deliberately skips `sessionAuth`/`csrfProtect` (a bearer token carried in a header, unlike a cookie, isn't subject to the cross-site request forgery CSRF exists to stop). Keys are `cwaf_`-prefixed 160-bit random tokens, shown once at creation; only a SHA-256 hex digest (`api_keys.key_hash`, unique-indexed for O(1) lookup) and a short display prefix are persisted — **SHA-256, not bcrypt**, unlike the admin password: bcrypt's deliberate slowness defends against brute-forcing a low-entropy human-chosen password, but these are high-entropy random tokens generated by the server, so bcrypt would only add latency to every API call for no security benefit (the same reasoning GitHub/Stripe/AWS apply to their own API tokens). `apiKeyAuth` reuses `loginLimiter` (`internal/ui/loginlimit.go`) as a second instance, `h.apiKeyLimiter`, keyed by client IP exactly like login brute-force protection, and mirrors `LoginPost`'s pattern of returning 429 immediately on the failure that crosses the threshold rather than waiting for the next request. A successful auth updates `last_used_at` in a spawned goroutine, throttled to once/minute per key, matching `autoban.ban()`'s existing "slow DB write off the hot path" pattern in this codebase — so, like that pattern, a key's last-use timestamp can very rarely lag behind a graceful shutdown's `db.Close()`; this is an accepted tradeoff, not new to this subsystem. Keys are all-or-nothing in v1 — full read-write, no read-only scope — managed from the Settings page's API Keys card (`CreateAPIKey`/`DeleteAPIKey` in `internal/ui/handlers.go`, session-authed like every other settings card). Endpoint coverage is services and IP rules (full CRUD, reusing the exact same `storage.DB`/`services.Registry`/`blocklist.IPBlocklist` calls the HTMX handlers use — see `AddService`/`AddIPRule` in `internal/ui/handlers.go` for the validation this mirrors) plus bans (read + manual ban/unban). There is no separate ban table: `GET /api/v1/bans` filters `ListIPRules()` to global (`app_name==""`) block rows, and `POST /api/v1/bans` writes via `AddIPRuleWithNote` with a `"Banned via API — "` note prefix — distinct from autoban's own `"Auto-banned — "` prefix so the two sources stay distinguishable in the IP Rules page's "Auto" badge (`IPRule.Auto()`); unbanning is just `RemoveIPRule`, identical to deleting any other IP rule.

**Bot protection** (`internal/security/bot/`, `internal/security/challenge/`, `internal/security/ja3/`): `bot.Analyze(r)` scores each request O(1) from headers only — scanner UAs (+10, `IsBot=true`), HTTP lib UAs (+3), missing headers (+1–2 each). Trusted SEO/social crawlers (Googlebot, Bingbot, Applebot, etc.) are detected first and bypass all scoring — `IsTrustedCrawler=true` skips the challenge gate entirely, regardless of per-service `bot_mode`. When a challenger is active, requests not carrying a valid bypass cookie are redirected to `/_cz/challenge` (JS proof-of-work, SHA-256 nonce, solved in <1s by real browsers). The challenge page also runs FingerprintJS (vendored MIT bundle `internal/security/challenge/fp.min.js`, served at `/_cz/fp.js` — vendored because ad blockers/Brave block the public CDN) concurrently with the PoW; the resulting browser visitor ID is HMAC-bound into the bypass cookie (`expiry.visitorID.sig`, legacy `expiry.sig` cookies stay valid) so every post-challenge request carries a tamper-evident visitor ID with no server-side session state. It's logged to `requests.visitor_id`, shown in the log detail modal, and drives the dashboard "Unique visitors" card — only clients that went through the challenge have one. The challenge page also probes for **automated-browser leakage** (`detectAutomation()` in `page.html`: `navigator.webdriver`, ChromeDriver `cdc_`/`$cdc_` arrays, `domAutomationController`, Selenium/Puppeteer/Playwright/PhantomJS/Nightmare globals, `HeadlessChrome` UA) and POSTs the hits to `/_cz/verify`; `ServeVerify` refuses to issue the bypass cookie when any recognized signal is present (`knownAutomationSignals` in `challenge.go`), so a headless scanner that solves the PoW natively still never earns a trusted session and stays stuck at the challenge — where the autoban scorer eventually IP-bans it for the repeated unsolved `bot_challenge` redirects. This is client-side detection (a bespoke attacker who reads the JS can strip the signals before POSTing); it defeats off-the-shelf browser-driven scanners, not a determined custom evasion. Challenge routes `/_cz/challenge` and `/_cz/verify` are always registered before the catch-all proxy route so Echo routes them correctly regardless of whether bot protection is currently enabled. They're method-specific (GET/POST), and Echo does not route HEAD to GET handlers — so any `/_cz/*` request that falls through to the catch-all (e.g. a scanner's `HEAD /_cz/challenge`) is 404'd by `Handle` before the challenge gate, never challenge-redirected (that 307 would point back at `/_cz/challenge` itself, an endless loop) and never proxied to a backend. TLS fingerprints — JA4 (`internal/security/ja4/ja4.go`, primary) and JA3 (`internal/security/ja3/ja3.go`, legacy) — are computed at TLS handshake time via `GetConfigForClient`, stored in each package's `sync.Map` keyed by `remoteAddr`, retrieved in the request handler (Cloudflare's `Cf-Ja4`/`Cf-Ja3-Fp` headers take precedence when present), and deleted by the `ConnState` hook in `startTLS` when the connection closes so the stores stay bounded. JA4 sorts ciphers/extensions before hashing, so handshake-order randomization doesn't change the fingerprint; JA3 is formally deprecated (see the `Deprecated:` notice in `internal/security/ja3/ja3.go`) and kept only for continuity with existing log data — never build new detection logic on it. Geo country lookups (`geo.Blocker.Check`) and ASN lookups (`asn.Lookup.LookupForConn`) reuse the same per-connection `sync.Map`-keyed-by-`remoteAddr` idea, but populated lazily on first request rather than eagerly at handshake time (a country/ASN lookup needs the resolved client IP, which isn't known until the request arrives, unlike a TLS fingerprint) — a keep-alive connection making many requests pays for one mmdb lookup instead of one per request. Each cached entry also records the client IP it was computed for and is discarded on a mismatch, since a reverse proxy that multiplexes several real clients over one pooled keep-alive connection to this origin (Cloudflare's connection pool to an origin behaves this way) would otherwise let one client's country/ASN leak onto another's request sharing the same `remoteAddr` — TLS fingerprints don't have this failure mode because Cloudflare's `Cf-Ja4`/`Cf-Ja3-Fp` headers are checked first and the local per-connection store is only a fallback for direct (non-CF) connections. Entries are deleted by the same TLS `ConnState` hook as JA3/JA4 when a connection closes; since that hook only exists on the TLS listener, both packages also run a background janitor goroutine that expires entries after `connCacheTTL` (10m) so plain-HTTP-only deployments — which have no `ConnState` hook at all — don't grow the cache unbounded. Per-service `bot_mode` — `inherit` / `always` / `off` — overrides the global setting; `always` challenges every non-trusted client regardless of score.

**Rate limiting** (`internal/security/ratelimit/`): the `Backend` interface (`Allow`, `TrackedIPs`, `Stop`) is satisfied by both `*Limiter` (in-process token bucket) and `*RedisBackend` (Lua-script atomic token bucket in Redis). The active backend is stored in `proxy.Handler.ratelimit` behind a `ratelimitMu sync.RWMutex`. `Limiter` has `Snapshot`/`RestoreFrom`/`StartPersistence(StateStore)` for SQLite write-back (every 10s), so token-bucket state survives restarts. `RedisBackend` fails open on Redis unavailability. `PingRedis(addr, password)` tests connectivity without creating a persistent client. Per-service limiters in `services.Registry` always use `*Limiter` (distribution not needed at the service level). The active backend choice (memory vs. Redis), Redis credentials, and the global limiter settings (meta keys `ratelimit_enabled`/`ratelimit_rps`/`ratelimit_burst` — off by default, opt-in from the Settings page) are stored in the DB `meta` table — not config.yaml; `buildRateLimit` in `main.go` reads them for both backends, so enforcement never differs between memory and Redis mode.

**Metrics** (`internal/notify/metrics/metrics.go`): Prometheus exposition format, served at `{admin.path}/metrics`. The route is registered on the admin Echo group (`internal/ui/handlers.go:Register`), so it sits behind the same **session-cookie auth + CSRF middleware** as every other admin page — it is _not_ open and _not_ HTTP Basic Auth, so a Prometheus scrape config cannot authenticate to it with a username/password alone (the README's "supports basic auth natively" note is inaccurate). `internal/proxy/handler.go` increments `IPBlockedTotal`/`GeoBlockedTotal`/`WAFBlockedTotal`/`RateLimitedTotal`/`BotChallengedTotal` at the exact decision point. `coraza_log_queue_depth` and `coraza_services_total` are `GaugeFunc`s that read live state on every scrape rather than via a polling goroutine.

**GeoIP database is bundled, not fetched.** `internal/security/geo/GeoLite2-Country.mmdb` is `//go:embed`-ed via `internal/security/geo/embedded.go` and used by `geo.New()` whenever `--geo-db` is empty, so the binary blocks by country out of the box with no MaxMind account or download step. Passing `--geo-db` with a real file path overrides the bundled copy.

**ASN lookup is also bundled.** `internal/security/asn/dbip-asn-lite.mmdb` (DB-IP Lite, CC BY 4.0) is `//go:embed`-ed via `internal/security/asn/embedded.go`. `asn.New()` opens it in-process — no external DB, no MaxMind account required. License attribution is in `THIRD_PARTY_NOTICES.md`.

**Hot-reload pattern**: three subsystems support live config changes without a server restart, all using the same `sync.RWMutex` swap pattern in `proxy.Handler`: `ReloadWAF(*waf.Engine)`, `ReloadBotProtection(*challenge.Challenger)`, `ReloadRateLimit(ratelimit.Backend)`. The Settings page save handlers call the corresponding `reloadX` callback passed into `ui.NewHandler` from `main.go`. The old value is stopped/discarded after the swap. `SIGHUP` rebuilds and swaps the WAF engine (the only way to pick up on-disk `rules_dir` edits without a restart; services already reload on every UI change). `SIGINT`/`SIGTERM` trigger graceful shutdown (`shutdownOnSignal` in `main.go`, 10s timeout) so in-flight requests complete and the deferred `db.Close()` drains the log queue before exit.

**Threat-intel auto-sync** (`internal/security/threatintel/worker.go`): `threatintel.New(db, reloadIPBL)` starts a background goroutine that periodically downloads external plain-text IP block lists, parses out IP/CIDR tokens (skipping `#`/`;` comments, capped at a 10 MiB read), and writes them to the DB. The `blocklist` package reads from that same store, so synced IPs take effect via the `reloadIPBL` callback with no restart. The UI's "sync now" button calls `Worker.SyncSource` (wired through `ui.NewHandler`) to fetch a single source on demand. `Worker.Stop()` is deferred in `main.go`.

**Webhook delivery** (`internal/notify/webhook/pusher.go`): `webhook.New(db.GetWebhookConfig)` returns a `Pusher` whose `Push(RequestLog)` is registered as the log fan-out hook via `db.SetWebhookFn` in `main.go`. Delivery is fully asynchronous over an internal goroutine (`run`/`deliver`) so a slow or unreachable webhook endpoint never blocks the SQLite log worker. `shouldSend` filters by the comma-separated event list in the singleton `webhook_config` DB row (id=1), managed from the admin UI.

**nginx-style access.log** (`internal/notify/accesslog/`, opt-in via `--access-log`): a fourth log fan-out hook alongside broadcast/webhook/autoban (`storage.DB.SetAccessLogFn`, called last in `runLogWorker` after the other three) — a flat nginx-combined-format text file for tooling that expects one (fail2ban, log shippers, `grep`/`awk`, `logrotate`), independent of the SQLite-backed admin UI. `accesslog.FormatLine(RequestLog) string` is the single source of truth for the line format and is reused by both the file writer and the dashboard's live terminal panel (below), so the two never drift. `storage.RequestLog` tracks neither response byte count nor the Referer header, so both render as the nginx convention `-` rather than a fabricated value — same placeholder already used for the identd/userid fields every combined-format line carries. Rotation is built in-house (`Writer.rotateLocked`: `access.log`→`access.log.1`→…→pruned beyond `--access-log-max-backups`) rather than pulling in a dependency, matching this project's existing "handle your own maintenance" pattern (`coraza-waf-mod prune` + a systemd timer, not an external tool, for the SQLite side). `Writer.Push` needs no locking for the write path itself — every call is already serialized through the single `runLogWorker` goroutine, the same guarantee every other fan-out hook relies on — the only mutex-guarded case is `Close` racing a write during shutdown; per the ordering `webhookPusher`/`banner` already establish, `defer accessLogWriter.Close()` is registered _after_ `defer db.Close()` in `main.go`'s source order so it still runs (LIFO) _before_ `db.Close()` while the queue drains.

**Live access-log terminal panel** (Logs page, live/non-history view only): a Table/Terminal toggle (`#view-table-btn`/`#view-terminal-btn` in `internal/ui/templates/logs.html`) switches between the existing structured HTMX row feed and a dark `bg-slate-900`/white-text monospace panel showing raw access-log lines, both fed by the same underlying stream. The terminal side subscribes to `GET /admin/access-log/stream` (`AccessLogStream` in `internal/ui/handlers.go`) — a third independent rendering of `ui.LogBroadcaster`'s broadcast stream (`LogsStream` emits HTML, `NotificationsStream` emits JSON, this emits a plain-text `accesslog.FormatLine` per event) — and works regardless of whether `--access-log` (the file) is set, since it reads the in-memory broadcaster, not the file. Only one SSE connection is ever open at a time: toggling the view (`setView` in `static/js/src/logs.js`) closes whichever stream is active and opens the other, rather than keeping both live — both endpoints read the same broadcast, so a second concurrent connection would buy nothing. Client-side, each line is appended as a text node (`.textContent`, never `insertAdjacentHTML`) — `Path` and `User-Agent` are attacker-controlled and must never be interpreted as markup — and the panel caps at the last 100 lines by dropping the oldest DOM node, mirroring how `LogBroadcaster`'s own ring buffer caps at 200 server-side. Lines don't soft-wrap (`whitespace-pre` + `overflow-auto`) — long lines scroll horizontally instead, like a real terminal. The panel preloads the last 24h of history on page load (`storage.DB.ListRecentRequestLogs`, `accessLogHistoryWindow`/`accessLogHistoryLimit` in `internal/ui/handlers.go`) rather than starting empty until new traffic happens to arrive; that query returns full `RequestLog` rows, not the lighter `LogRow` the table view uses, since `FormatLine` also needs `Query`/`Proto` that `LogRow` doesn't carry. Auto-scroll-to-newest only fires while the user is already at (or within `accessLogScrollThreshold`px of) the bottom — tracked via a `scroll` listener on `#access-log-wrapper`, independently of the existing manual Pause button; either one stops auto-scroll, since scrolling up to read history shouldn't require also remembering to hit Pause.

**Daily report email** (`internal/notify/mailer/`): `mailer.NewReporter(db)` (started in `main.go`) sleeps until just past local midnight, then emails a summary of the completed day — total/blocked/403 counts plus a per-stage breakdown from `storage.GetDailyReport`. Delivery is **Cloudflare Email Service only**: the endpoint (`smtp.mx.cloudflare.net:465`, implicit TLS — no STARTTLS), the literal `api_token` username, and the sender address (`mailer.Sender`) are hardcoded constants in `internal/notify/mailer/mailer.go`; the only configurable values are the recipient list and the Cloudflare API token, stored in the `meta` table (`storage.EmailConfig`) and entered via the Settings page — **the token is never shipped in the repo, binary, config.yaml, or install.sh**, and the save handler never echoes it back (blank field keeps the stored token). Sends are idempotent per day via the `email_report_sent_for` meta key, so restarts around midnight neither duplicate nor skip a report (a failed send is _not_ marked, so the next startup/tick retries). `Reporter.SendNow` (wired into `ui.NewHandler`) powers the "Send test report" button and works with the schedule disabled. The HTML body mirrors the admin dashboard theme with inline styles and nested tables; Gmail strips SVG, so the logo badge is rebuilt as HTML (green rounded square + three filter bars). `net/smtp.PlainAuth` works over implicit TLS because `smtp.NewClient` detects the `*tls.Conn` and marks the session TLS.

**Automatic IP banning** (`internal/security/autoban/`): `autoban.New(db, reloadIPBL, notify)` registers `Banner.Record` as a second log fan-out hook via `db.SetAutobanFn` (same pattern as the webhook pusher — both run on the single log-worker goroutine, so `Record` is in-memory only). Each scored event adds points per client IP in a sliding window (critical CRS rule 930xxx–934xxx/941xxx–944xxx = 5, other WAF rule = 2, `rate_limited` = 1, unsolved `bot_challenge` redirect = 1; already-blocked `ip_blocked*`/`geo_blocked*` traffic never scores, or banned IPs would re-trigger forever). Challenge redirects are the one non-`Blocked` event class that scores — the challenge gate runs _before_ the WAF in `Handle`, so a scanner stuck in the challenge loop never produces WAF-blocked rows and would otherwise probe forever without ever being banned; clients that solve the challenge stop producing `bot_challenge` rows and accumulate nothing. Crossing the threshold writes a **permanent global block rule** with a reason in the new `ip_rules.note` column (`AddIPRuleWithNote`; rows with an `Auto-banned` prefix render an amber "Auto" badge on the IP Rules page), hot-reloads the blocklist, and emails the admin via `mailer.Reporter.SendBanAlert` (silently skipped when email alerts are off — a ban must never depend on email config). The slow path (DB write + reload + SMTP) runs on a spawned goroutine; a `banned` dedupe map is marked synchronously first so queued events can't double-ban. Private/loopback/link-local IPs are never banned, and an existing admin rule (allow _or_ block) always wins (`GetIPRuleType` check inside `ban`). Config (`storage.AutobanConfig`: enabled/threshold/window, default on at 10 points per 10 min) lives in meta keys `autoban_*`, edited from the "Automatic banning" card on the IP Rules page, applied live via the `reloadAutoban` callback (`Banner.ReloadConfig`) passed to `ui.NewHandler`.

**Unified per-IP threat score** (`internal/security/threatscore/`, issue #12): `threatscore.New(db, banner.Score)` registers `Scorer.Record` as a third log fan-out hook via `db.SetThreatScoreFn`, wired in `main.go` right after `db.SetAutobanFn` so autoban has already folded the current event into its history by the time `Record` reads it. `Record` also caches the score it just computed in an in-memory `map[string]int` (bounded by a `janitor` goroutine mirroring `ratelimit.Limiter`'s bucket eviction, but with a much longer 30-minute idle TTL — a threat score is standing reputation, not a short-lived rate-limit window), read back synchronously via `Scorer.CurrentScore(ip) int` — the hot-path lookup that issue #16's adaptive enforcement (below) uses instead of a SQLite query per request. Because `Record` runs asynchronously after its triggering request was already logged, `CurrentScore` always reflects an IP's _previous_ requests, never the one in flight — the same one-request lag `autoban`'s ban already has. The composite score (0–100) sums five capped components: autoban's current point total for the IP via a new exported `Banner.Score(ip) int` accessor (up to 40 — deliberately not built by reusing `pruneEvents`, which compacts its slice in place via `evs[:0]` and would corrupt `Banner.hist`'s backing array if called read-only without reassigning the result), the bot-analysis score already on the log entry (up to 20, no new work), an ASN/hosting classification (flat 15) from a small hardcoded heuristic in `asnclass.go` — the bundled DB-IP ASN Lite data has only a number and an org-name string, no type field, so this is a non-exhaustive ASN allowlist plus an org-name keyword fallback, deliberately excluding Cloudflare's own ranges since the bot/challenge subsystem already has a separate trusted-crawler path for that traffic — a geo-risk flag (flat 10) that reuses the existing `geo_rules` "block" entries as the risk set rather than adding new admin-facing config (refreshed via `Scorer.ReloadGeoRules`, called alongside every existing `geoBl.Reload` call site so a geo-rule edit updates both with no restart), and a JA4 repeat-offender component (up to 15) backed by a new `ja4_reputation` table (`hits`/`blocked_hits` per fingerprint, `db.BumpJA4Reputation`) since the `ja4` package's own store is connection-scoped and forgets fingerprints the moment a connection closes. Scores persist in `ip_threat_scores` (one row per IP, upserted, with the per-component breakdown so the UI can show _why_ an IP scored what it did) — a new table via `CREATE TABLE IF NOT EXISTS`, not an `ALTER TABLE` migration, following the `api_keys`/`rate_state` precedent for bolt-on features. Surfaced on the log detail modal (`#ld-threat-section`, hidden until an IP has a recorded score) and as a "Score N" badge per row on the IP Rules page (`GetIPThreatScores` bulk-fetches the current page's scores in one query rather than one lookup per row).

**Threat-score-driven adaptive enforcement** (`internal/security/adaptive/`, issue #16): consumes the threat score above to dynamically scale the global rate limit and force a bot challenge — the two levers that were cheap and fully reversible with the current architecture. **Deliberately excludes Coraza paranoia-level tiering** (the issue's third proposed lever): paranoia level is baked into the compiled `*coraza.WAF` engine at build time via CRS's `SecAction ...setvar:tx.blocking_paranoia_level=N`, with no per-transaction override in Coraza's Go API, so supporting it would mean maintaining multiple fully pre-built engines in memory (each holding the whole OWASP CRS ruleset) and picking one per request — real memory cost and doubled `ReloadWAF`/SIGHUP rebuild time on a single-binary WAF sized for small deployments. Tracked as a follow-up issue instead of built silently. `adaptive.Policy` is policy only, no mechanism: `Decide(score int) Decision` is a pure function (no I/O) returning a `RateScale` (1.0 = unchanged) and `ForceChallenge` bool, config-reloadable via `Policy.ReloadConfig()` — same `New(db)`/no-arg-self-reads shape as `autoban.Banner.ReloadConfig`. Two independently configurable thresholds: `HighRiskThreshold` (score ≥ this tightens the rate limit to `HighRiskRateScale`, e.g. 0.3) and a separate, stricter `ForceChallengeThreshold` (only past this does a high-risk client also get force-challenged) — so tightening and challenging don't have to happen at the same score. `LowRiskThreshold` relaxes the limit via `LowRiskRateScale` (e.g. 1.5) and never forces a challenge. **Disabled by default** (`storage.DefaultAdaptiveEnforcementConfig`), unlike autoban's enabled-by-default — this is a brand-new mechanism riding on a heuristic score (the ASN classification above can misfire) and includes a security-_loosening_ action (relaxing limits for low-risk IPs), so an admin should see real scores on the IP Rules page before opting in. `ratelimit.Backend` gained `AllowScaled(ip string, scale float64) Result` (both `*Limiter` and `*RedisBackend` implement it; `Allow` is now a thin wrapper calling it with `scale=1.0`) — a `scale <= 0` or a resulting burst below 1 floors to 1 token, so a scoring bug can never fully lock out traffic, matching the issue's reversibility requirement. `proxy.Handler.Handle()` computes `score := h.scorer.CurrentScore(clientIP)` and `decision := h.adaptive.Decide(score)` once per request (both nil-safe — a `*threatscore.Scorer`/`*adaptive.Policy` field left `nil`, e.g. in tests that don't need this feature, behaves as if disabled rather than panicking, the same nil-receiver-guard pattern `asn.Lookup.Lookup` already uses) and applies `decision.RateScale` at the _global_ rate-limit gate only (per-service limiters are untouched — out of scope) and ORs `decision.ForceChallenge` into the bot-challenge gate alongside the existing per-service `bot_mode == "always"` and `botAnalysis.Score >= ch.Threshold()` checks. Transparency reuses the existing free-form `Action`-string convention rather than adding new columns/metrics: a block/challenge caused _only_ by adaptive enforcement is logged as `"rate_limited:adaptive"` / `"bot_challenge:adaptive"` instead of the plain string. Every place that used to exact-match `action = 'bot_challenge'` or `'rate_limited'` (autoban's own scoring, the webhook pusher's event-category filter, `GetBotStats`/`GetDailyReport`'s stat queries, the dashboard toast's action-to-badge mapping in `static/js/src/toast.js`) had to switch to a prefix match (`LIKE 'bot_challenge%'` / `strings.HasPrefix`) — the same fix `ip_blocked`/`geo_blocked` needed for their own reason-suffixed variants, so autoban still scores an adaptively-forced challenge redirect exactly like a normal one. Config lives in meta keys `adaptive_*`, edited from the "Adaptive enforcement" card on the IP Rules page (next to Autoban, since both act on the same per-row threat score shown there), applied live via `reloadAdaptive` (`Policy.ReloadConfig`) passed to `ui.NewHandler`.

**SQLite date/time gotcha**: never use SQLite's `date()`/`strftime()` functions on the `requests.ts` column — `modernc.org/sqlite` stores `time.Time` using Go's `.String()` format (`"... +0000 UTC"`), which SQLite's date functions cannot parse and silently return NULL for. All date bucketing (hourly traffic, retention cutoffs) is done in Go using plain `>=`/`<=` comparisons against `ts`. See `internal/storage/db.go`'s `GetHourlyTraffic` for the canonical pattern.

**Security headers** (`internal/proxy/security.go`): `SecurityMiddleware()` is registered globally on the Echo instance (applied before routing), adding standard browser-hardening headers (`X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`, `COOP`) plus HSTS on TLS connections and an `X-WAF-Engine` identification header (`X-Protected-By` was removed to reduce WAF fingerprinting via response headers). Registered once in `main.go`, so blocked responses, admin UI pages, and proxied responses all carry the same headers.

**`proxy.responseWriter` must proxy Hijack and Flush**: the status-capturing wrapper in `internal/proxy/handler.go` wraps `http.ResponseWriter` to intercept the status code for logging. Go does not automatically promote interface methods from the embedded concrete value, so `http.Hijacker` and `http.Flusher` must be explicitly implemented and delegated — without them, WebSocket upgrades (e.g. Vite HMR) fail with "can't switch protocols using non-Hijacker ResponseWriter".

**No config file, by design.** The running server takes zero YAML/JSON/TOML config — every bootstrap setting (listen addresses, TLS binding, WAF rules dir, DB path, retention) is a CLI flag (see the flag table above), and every runtime knob — bot protection settings, Redis config, ACME email, per-service bot/TLS/rate-limit overrides — lives in the SQLite `meta` table or per-service columns, managed from the admin UI with no restart. `config.yaml` and `deploy/config.yaml.example` still exist as files in the repo and are referenced by some comments/README text as a historical artifact of an earlier YAML-driven design, but no Go code parses them (`grep -rn "yaml\." --include="*.go"` outside tests turns up nothing) — treat any doc or comment that says a setting lives "in config.yaml" as describing a CLI flag or DB-backed UI setting instead, and prefer flags/DB when in doubt.

**Module layout**: every implementation package lives under `internal/` (the Go-standard signal that these aren't a library for external import — `main.go`/`assets.go` stay at repo root since `assets.go`'s `//go:embed static/js/dist/*`/`//go:embed static/imgs/*` can't reach outside its own directory, so moving them under `internal/`/`cmd/` would force `static/` to move too). Grouped by concern, not left flat:

- `internal/config/` (flag-driven `Config` struct + `Defaults()`, no file parsing) and `internal/storage/` (SQLite access, schema migrations, async log queue, `StateStore` impl for rate-limit persistence) sit as direct `internal/` children rather than inside a group — both are near-zero-dependency foundations nearly everything else imports, and `storage` in particular is the most-imported package in the module, so keeping its path short pays off repeatedly.
- `internal/security/` bundles every signal, mitigation, or blocking mechanism the proxy pipeline consults: `waf/` (Coraza engine wrapper), `blocklist/` (IP rules, exact-match), `geo/` (GeoIP2 country blocking, bundled MaxMind GeoLite2, per-connection lookup cache), `asn/` (ASN/org lookup from bundled database, same per-connection lookup cache), `ratelimit/` (token-bucket `Backend` interface + in-process `Limiter` + `RedisBackend`), `bot/` (header-based signal scoring + trusted-crawler list), `challenge/` (JS PoW challenge page + HMAC token/cookie), `totp/` (RFC 6238 one-time passwords for admin-login 2FA — used by `internal/ui`, not the proxy pipeline), `ja3/` (legacy JA3 TLS fingerprint + per-connection sync.Map store), `ja4/` (JA4 TLS fingerprint, same store pattern; primary fingerprint in logs/UI), `autoban/` (scores blocked events per IP, writes permanent block rules + email alerts), `threatintel/` (background sync of external IP block lists into the DB), `threatscore/` (unified per-IP composite score read model — distinct from `threatintel` despite the similar name), `adaptive/` (threat-score-driven rate-limit/challenge policy decisions — no enforcement mechanism of its own, just a pure `Decide` function).
- `internal/proxy/` (the full request pipeline) and `internal/services/` (DB-backed routing registry + TLS) stay direct `internal/` children, siblings rather than nested under an invented parent for just the two of them.
- `internal/notify/` groups the outbound-reporting mechanisms: `mailer/` (SMTPS sender + midnight daily-report scheduler), `webhook/` (async event delivery to a configured webhook endpoint), `metrics/` (Prometheus instrumentation), `accesslog/` (opt-in nginx-combined-format access.log writer with in-house rotation).
- `internal/ui/` (admin dashboard handlers + templates) stays a direct child — already a large, distinct subsystem with its own `templates/` subtree.
- `tools/minify/` (build-time JS minifier, not shipped in the binary) is not under `internal/` — it's a standalone `go:generate`-invoked tool, not a package the running server imports.

**User-facing docs**: `docs.md` at the repo root is a large consolidated documentation build, assembled in Docusaurus sidebar order from a `docs/` tree that lives in a separate docs-site repo — the `docs/` directory it calls canonical does not exist here, so treat `docs.md` as a snapshot that can lag the code (like the README's known-inaccurate metrics-auth note), never as authoritative over the code or this file.

## Security & configuration tips

Never commit real admin credentials, TLS private keys, GeoIP databases, or a production `waf.db` file. Admin credentials are seeded via `coraza-waf-mod setup` (password on stdin, never a flag or file) — nothing secret should ever land in `config.yaml`/`deploy/config.yaml.example`, which the server doesn't read anyway (see "No config file, by design" above).

**Secrets-at-rest encryption** (`internal/storage/secretenc.go`, opt-in via `--db-key-file`): the live credentials stored in `waf.db` — challenge-cookie HMAC secret, TOTP active/pending secrets, Cloudflare `email_token`, `redis_password`, and the `webhook_config.secret` column — are sealed with AES-256-GCM (`enc:v1:` + base64(nonce‖ciphertext)) when a key file is configured; everything already safe at rest (bcrypt admin password, SHA-256 API-key and TOTP-backup-code digests) and everything non-secret (request logs, rules, services) stays plaintext by design, deliberately NOT whole-file encryption (that would need SQLCipher/CGO or a driver swap — rejected to keep `CGO_ENABLED=0` cross-compiles). The key is SHA-256 of the key file's whitespace-trimmed contents (any ≥32-byte high-entropy file works, e.g. `openssl rand -hex 32 > db.key`; a trailing newline doesn't change the key). Reads are format-dispatched on the prefix, not mode-dispatched: plaintext values pass through with or without a key (so enabling is a transparent in-place migration via `EnableSecretEncryption` in `main.go`, no manual step), while a sealed value read without/with the wrong key errors loudly — never silently-empty config, and `GetOrCreateChallengeSecret` specifically must never regenerate on that error (it would invalidate every outstanding bypass cookie and mask the misconfiguration; `TOTPEnabled` similarly reads the raw value so a key problem can't fail login open to password-only). Any new secret-bearing meta key MUST be added to `secretMetaKeys` and accessed only via `getSecretMeta`/`setSecretMeta` (`getMeta`/`setMeta` would write it plaintext under an active key). `install.sh` generates `/etc/coraza-waf-mod/db.key` (0400, service-user-owned, never regenerated on upgrade) and passes the flag in its systemd unit — deliberately outside `/var/lib/coraza-waf-mod`, so exfiltrating the data dir or a Settings backup (a byte-for-byte `VACUUM INTO` copy, sealed values included) doesn't also yield the key. Full-disk needs (log PII, `certs/` private keys) are LUKS/fscrypt territory at the deployment layer, not app code.

## Distribution

Native binary is the primary distribution target. A `Dockerfile` (multi-stage: `golang:1.25-alpine` builder → `scratch` final, `CGO_ENABLED=0`) and `docker-compose.yml` exist for container-based local development but are not used in production. `make dist` cross-compiles `linux/amd64`/`linux/arm64`/`windows/amd64` with `CGO_ENABLED=0` (works because every dependency, including the SQLite driver, is pure Go). `deploy/install.sh` is a one-line `curl | sudo bash` installer (Linux only): downloads + SHA256-verifies the binary from GitHub release assets, creates a dedicated non-root system user with only `CAP_NET_BIND_SERVICE` (not root) for binding privileged ports, and writes its own systemd units via heredoc (`ExecStart=... --db ... --certs ... --retention 30`, flags only) — it does **not** copy the checked-in `deploy/coraza-waf-mod.service` / `deploy/coraza-waf-mod-prune.{service,timer}` files, which are stale leftovers from an older `config.yaml`-positional-arg invocation style (`ExecStart=... /etc/coraza-waf-mod/config.yaml`) and don't reflect what actually gets installed; trust `install.sh`'s heredocs, not those three files, if the two disagree. Check status with `sudo systemctl status coraza-waf-mod`.
